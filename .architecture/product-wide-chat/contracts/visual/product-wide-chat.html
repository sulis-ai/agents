Per-product chat — one conversation per product, switched like the board

Each product has its own chat. You switch products with the same top-left switcher you already use for the board — and when you do, the board and the chat both move to that product together. Pick "Clinics" and you see the Clinics board next to the Clinics conversation. Which AI powers the chat (Claude or Antigravity) is still picked inside the chat. Rendered with the cockpit's real colours and fonts, light and dark.

The correction this pass makes. Two earlier passes mis-scoped the chat — first as one change, then as one global chat for everything. Both were wrong. The chat is scoped per product, and you move between products' chats with the existing product switcher — exactly the way you already move the board. One switch moves both. Each product keeps its own conversation history. The Claude↔Antigravity choice still lives at the foot of the composer.

3 · The "All products" and "Unassigned" scopes — what the chat shows there
Design decision — your call What should the chat be when you're not inside a single product? The switcher has two scopes that aren't a single product: All products (everything) and Unassigned (work not yet filed under a product). The recommended answer, shown below:
  • All products → an overview chat. One cross-product conversation with Sulis: "what needs me across everything?", "what shipped this week?". It can start work, and because no single product is selected, it asks which product the new work belongs to before it files the card. It does not blend the per-product histories into one stream — those stay separate.
  • Unassigned → a triage chat. The conversation here is about the loose work that hasn't been filed yet: "file these under a product", "what's this orphaned change?". Its natural job is sorting, not building.
This keeps every chat to one clear job and avoids a single firehose. Tell me if you'd rather "All products" be a quieter read-only overview with no chat at all — that's the main fork here.

4 · The three states every surface must show — loading · empty · error

The chat dock in each of its three honest states (shown here for a product chat). Empty is a product's first-run / fresh thread. Loading is while that product's history streams in after you switch to it. Error is when the chat can't be reached — said plainly, with one clear retry, and never dressed up as an answer. 5 · Honest phasing — smallest coherent first surface vs the fuller vision

First surface build first

  • One chat per product, swapped by the existing switcher together with the board.
  • The chat header echoes the active product's tile (whose chat am I in).
  • Free-text + suggestion chips + the slash hint (the composer you already have).
  • The agent picker at the composer foot — Claude (default) ↔ Antigravity — naming the active agent.
  • "Talk → a change card appears on this product's board", with a click-through link.
  • The confirm gate before starting work; honest working/idle status; the three states.
  • "All products" overview chat (asks which product new work belongs to). Keep it simple.

The fuller vision later

  • The "Unassigned" triage chat (sort loose work into products) — second, once All + per-product land.
  • Per-product memory transparency (see and correct what Sulis remembers about each product).
  • Multiple named threads within one product, each showing "agent working here".
  • An "Auto" agent option that picks Claude vs Antigravity per task.
  • Mid-run pause / stop on a running agent from the chat.
Accessibility, checked against the real tokens (light + dark). Every state carries a word, not colour alone — "Working…", "Started", "Powered by Claude", "Clinics chat", "I couldn't reach this chat". The product tile reuses the switcher's never-colour-alone shapes (monogram letters / grid-of-dots for All / dashed line for Unassigned), so scope is legible without relying on hue. The switcher and agent picker are real menu buttons (aria-haspopup="menu" / aria-expanded) opening role="menu" lists with menuitemradio + aria-checked rows, full keyboard parity, ≥30px rows with a visible focus ring. Body text uses the cockpit's verified token pairings, all clearing WCAG AA in both themes. Motion (streaming caret, skeleton shimmer) has a prefers-reduced-motion fallback.
