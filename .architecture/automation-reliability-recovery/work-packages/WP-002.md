# Provider-neutral classifier

## Why

The classification *policy* must be provider-neutral so a second provider
needs only a new thin adapter, not a change to the shared layer (spec; ADR-003).
This WP is the innermost new layer: a pure function over the existing
`EventError`, depending on `events.py` alone. It is the producer in the
WP-001 contract.

## What changes

- `classify(error, adapter_hint) -> RecoveryClass` (the home is `classifier.py`
  per ADR-003).
- Depends on `events.py` and the `RecoveryClass` vocabulary (WP-001) **only** —
  no adapter import.

## How

Per ADR-003 and the contract truth table:

1. If `adapter_hint` is a `RecoveryClass`, use it (the provider knows better).
2. Otherwise apply the neutral default from `category` alone:
   - `protocol` → `TRANSIENT_BLIP` (transport wobble; matches the
     `ProtocolError` docstring already in `events.py`)
   - `internal` → `DEAD_END` (a bug; the model says log + escalate, don't retry)
   - `expected` → `DEAD_END`, **except** `NOT_AUTHORIZED` → `LOGIN_EXPIRED`
     (the one `expected` code with a neutral login meaning)
3. Unknown future code with no hint → the category default → safe direction
   (dead-end for expected/internal). The function is **total** and never raises.

Boring and explicit: a flat mapping, no reflection, no string-keyed dispatch
into provider vocabulary (the `"401"` magic-string interpretation lives in the
Claude adapter, WP-006 — never here).
