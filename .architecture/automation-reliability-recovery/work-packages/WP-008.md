# Observability + end-to-end recovery integration test

## Why

The unit WPs prove each piece in isolation against fakes. This WP is the CF-07
conformance close: it drives the **wired** manager (WP-007) end-to-end and
asserts the acceptance-critical, founder-visible behaviour — that every
classification and recovery action is observable in the existing error/event
stream (the spec's "a later reviewer can see why a run was retried / abandoned
/ paused"). It is the integration child that closes the contract-first arc.

## What changes

- `plugins/sulis/scripts/tests/integration/test_session_manager_recovery_e2e.py`
  — new integration test driving recovery through the real wired manager with a
  scripted child + fake clock + fake re-auth ticket (the existing
  integration-test style; MEA-09 — real manager state, real log, deterministic
  child, no real sleep).
- No production code (test-only).

## How

Per TDD §4.2 and the contract's emitted-Event shape table. For each recovery
class, drive the wired manager to the failure, then read back via
`read(follow=True)` and assert the emitted `Event` matches the contracted
shape (kind / category / code / message-carries). The login-expired test
asserts the `NOT_AUTHORIZED` Event carries the re-login link and that resume
re-runs the step (no result Event before the resumed turn produces one — no
fabricated completion).

The **real** login-expiry → real re-auth → real resume round-trip against a
live `claude` with an expired credential is **out of scope here** — it cannot
bootstrap in CI and is the deferred manual need `live-reauth-resume-claude`
(ARCH.yaml; TDD §4.3). This WP covers the logic against the fake clock + fake
ticket + the recorded-resume discipline the lifecycle already tests.
