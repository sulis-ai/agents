# Adapter seam extension — classify_failure + reauth

## Why

The *detection* of which raw failure means login-expiry vs blip vs dead-end is
per-provider and must live behind the existing `ProviderAdapter` seam — the
"only agent-specific surface" (ADR-003). This WP extends that Protocol with
exactly two thin, defaulted, additive methods and nothing more (spec: "do not
fork or widen beyond the thin detection + re-auth additions"). It is the seam
the Claude implementation (WP-006) and the driver (WP-005) bind to.

## What changes

- `_session_manager/adapter.py` — add to the `ProviderAdapter` Protocol:
  - `classify_failure(self, error: EventError) -> RecoveryClass | None`,
    **defaulted to `None`** so a future adapter that doesn't override it stays
    safe (defer to the neutral classifier).
  - `reauth(self) -> ReauthTicket` — begin re-auth, return the WP-001
    `ReauthTicket`. Only called on `LOGIN_EXPIRED`.

## How

Per ADR-003. This is `EXPAND-Extend` of a Protocol **we own** — the
Stripe-rule discriminator (the public face is the session manager's own seam,
not a vendor's), so it is EXPAND, never SUBSTITUTE-Wrap. The additive,
defaulted shape mirrors the existing `io_mode` / `brief_change_id` precedent
already in this file: new optional surface, existing signatures byte-unchanged.

Per EP-07 / TDD §2.2: this **adds** methods, it does not restructure existing
behaviour, so it is **not** a structural change requiring a characterisation
test. The conformance test below is the additive proof, not a refactor guard.

## Tests (Red → Green → Blue)
