# Claude provider detection + re-auth

## Why

This is the thin, per-provider half of the seam: the **only** place Claude's
HTTP-status vocabulary is interpreted (ADR-003). The Claude adapter already
collapses provider HTTP failures into an `expected` error whose `code` is the
raw status (`"401"`, `"429"`, `"400"` …) via `_error_payload`; this WP maps
those raw codes to a `RecoveryClass` hint and provides the re-auth trigger.
It is the one WP that exercises a provider (acceptance: detection + re-auth
exercised per-provider).

## What changes

- `_session_manager/adapters/claude.py` — implement the two methods the
  WP-004 Protocol now defines:
  - `classify_failure(error)`: `"401"`/`"403"` → `LOGIN_EXPIRED`; `"429"` and
    connection-reset shapes → `TRANSIENT_BLIP`; `"400"` and other deterministic
    declines → `DEAD_END`.
  - `reauth()`: return a `ReauthTicket` (WP-001) carrying the Claude re-login
    link + a completion handle.

## How

Per ADR-003. `EXPAND-Extend` on an adapter behind the seam we own (not a wrap).
The mapping is a flat, explicit lookup over the raw `api_error_status` — no
shared-layer leak. Reuse the existing Claude error fixtures under
`plugins/sulis/scripts/tests/fixtures/session_manager/claude/` as the recorded
`EventError` seeds (referenced, not duplicated — TDD bootstrap-from-zero).

The real `reauth()` round-trip (live `claude` + a genuinely expired credential)
is verified **manually** on the founder machine — deferred need
`live-reauth-resume-claude` (ARCH.yaml / TDD §4.3). This WP's automated tests
cover the mapping logic and that `reauth()` returns a well-formed ticket; they
do not perform a live login.
