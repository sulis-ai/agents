Platforms

platforms

Creating and managing the isolated environments customers run on.

What it does

1
An admin sets up a new platform
They send a name and a slug for the new platform.
→
2
The product creates the platform
It stores the details and assigns an ID.
→
3
Here's exactly what comes back
The product returns this shape, every time.

A platform, with these fields:

idstringA unique identifier for the platform.
namestringThe platform name, exactly as it was entered.
slugstringThe short URL-safe handle for the platform.
createddate-timeWhen the platform was created.
  • create An admin can set up a new platform. Operation platform/create-platform create_platform Needs sign-in Runs in the background
    Permission: lets you create new platforms platform.platforms:create
    Who can do this: Platform administrators and account owners
  • read The product can fetch a platform by its ID. Operation platform/get-platform get_platform Needs sign-in
    Permission: lets you view a platform's details platform.platforms:read
    Who can do this: Any signed-in member of the platform
  • list Anyone can list the public platforms. Operation platform/list-platforms list_platforms No sign-in
    Who can do this: Anyone — no sign-in required
  • rename An admin can rename a platform. Operation platform/rename-platform rename_platform being retired — until 2026-09-30 Use “Configure Platform” instead. This action is being retired on 30 September 2026. Needs sign-in
    Permission: lets you change a platform's settings platform.platforms:update
    Who can do this: Platform administrators

The things that exist, and the states they move through

Platform
An isolated environment for one customer's application.
A platform moves through these states:
draft → active → suspended

The rules it follows

  • PlatformA slug can't change once the platform is created.
  • PlatformTwo platforms can't share the same slug.
  • PlatformA suspended platform can't be set back to draft.

Languages

Available in: English (default) Spanish

Step-by-step journeys

Onboard a new customer
  1. An admin creates a platform with a name and slug.
  2. The admin configures the platform's settings.
  3. The admin invites the customer's first users.
Done when: the platform is active and at least one user has been invited.

What each action changes

  • create sets aPlatform→draft
  • activate moves aPlatform fromdraft→active
  • suspend moves aPlatform fromactive→suspended

A real example

When the admin sends —
POST /platforms
{
  "name": "Acme Production",
  "slug": "acme-prod"
}
— the product returns:
201 Created
{
  "id": "plt_01H8X...",
  "name": "Acme Production",
  "slug": "acme-prod",
  "created": "2026-05-29T10:24:00Z"
}

When something goes wrong

  • DUPLICATE_SLUG A platform with this slug already exists. Choose a different slug.
  • VALIDATION_ERROR One of the fields didn't meet requirements. Check the name and slug and try again.

Notifications

notifications

Sending and tracking the messages that go out to a platform's members.

What it does

  • send An admin can send a notification to a platform's members. Operation notification/send-notification send_notification Needs sign-in Runs in the background
    Permission: lets you send notifications to members platform.notifications:send
    Who can do this: Platform administrators
  • list A member can see the notifications sent to them. Operation notification/list-notifications list_notifications Needs sign-in
    Permission: lets you read your own notifications platform.notifications:read
    Who can do this: Any signed-in member of the platform
