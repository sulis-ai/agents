**Approach.** REINFORCE-Harden over existing durable wiring. Root cause: SessionManager.open() runs _attach_durable_sink (real ~/.sulis store mkdir+read+write) synchronously under the registry lock BEFORE start_pumps and before the open result returns; it is NOT isolated against exceptions OR hangs, so on the cockpit-e2e CI runner (real ~/.sulis, not the test's isolated temp dir) the open round-trip never completes. Fix: bound + try/except-isolate the attach-time durable I/O AND the respawn reseed so a store that raises, hangs, or hits a restricted FS degrades to no durable sink while the live session opens unchanged. Key risk: hang-isolation needs a bounded timeout off a worker thread (signal.alarm is main-thread-only and the daemon serves on worker threads).
