# Agent-Consumable SDK Documentation Specification

**Version:** 0.1.0
**Date:** 2026-05-21
**Status:** Specification — companion to the SDK spec at
[`agent-consumable-sdk-spec.md`](agent-consumable-sdk-spec.md)
**Audience:** Agents producing documentation for SDKs built per the SDK
specification
**Standards applied:** CP-01 (Convention Preference), Diátaxis framework,
Critical Thinking Standard (MECE / PG / BI)

---

## What this spec is for

When an agent builds an SDK per `agent-consumable-sdk-spec.md`, it ships
the typed clients (Python, TypeScript) and the MCP server. That's the
runtime surface. **This spec covers the documentation that accompanies
it** — the writeable artefacts that turn a generated SDK into a
usable one.

The motivation: a typed SDK without docs is a maze where every
operation is reachable but the agent (and the human) doesn't know which
to call, when to call it, or what to do when it fails. The docs are the
map.

---

## Primitives (PG — primitive grounding)

Documentation has been a solved category for a while. Per CP-01, the
established convention is **Diátaxis** — Daniele Procida's framework
that splits docs into four MECE quadrants based on what the reader
needs in the moment.

### The four documentation primitives

| Primitive | Reader's state | Doc's job | Example |
|---|---|---|---|
| **Tutorial** | "I'm new; teach me" | Hand-hold from zero to first success | "Build your first agent in 10 minutes" |
| **How-to guide** | "I know what I need; show me how" | Recipe for one specific task | "How to handle rate limits" |
| **Reference** | "I'm in the middle of work; what does X do?" | Exhaustive, dry, fact-shaped | API method docs, configuration table |
| **Explanation** | "I want to understand" | Conceptual model, why-shaped | "How retries work; why we chose this default" |

**MECE check:** Every doc artefact fits exactly one quadrant. A doc
that tries to be more than one ("teach me how to use this while
explaining why") is the failure mode Diátaxis was designed to prevent.

**Independence check (PG-02):** Could two quadrants collapse into one?
No — they serve different reader needs and use different writing
conventions. A tutorial is hand-held and friendly; a reference is dry
and authoritative; they don't substitute.

**Irreducibility check (PG-04):** Could a quadrant split further? Yes,
within each (e.g., per-language tutorials, per-transport how-tos). But
at the *primitive* level there are four, not more.

### Mapping the user's missing-docs table to the primitives

The user surfaced 15 documentation topics that an SDK should have but
doesn't. Each maps to a primary Diátaxis quadrant:

| Topic | Primary quadrant | Secondary needs |
|---|---|---|
| Getting started (each language) | Tutorial | Reference for setup |
| Configuration (timeout, retry, throttle, base URL) | Reference | Explanation for defaults |
| Pagination patterns | How-to | Reference for the pagination types |
| Error handling — when to catch which class | How-to | Explanation for the category model |
| Identity-mismatch wart | Explanation | How-to for handling it |
| Raw-response access | How-to | Reference for the API |
| Timestamp handling | Reference | Explanation for the format choice |
| Async vs sync decision guide | Explanation | How-to per branch |
| TypeScript notes (browser, custom fetch, AbortSignal) | How-to + Reference | — |
| MCP setup per client (Claude Desktop, Cursor) | Tutorial | Reference for config |
| Full MCP tool reference | Reference | (autogenerated from schema) |
| Recipes / cookbook | How-to | — |
| Troubleshooting / FAQ | How-to (symptom-first) | Reference for error catalogue |
| Versioning policy + migration notes | Explanation (policy) + Reference (per-version diff) | — |
| API reference per language | Reference | (autogenerated from schema) |

All 15 topics fit. The 4 primitives cover the surface MECE. Below those,
we layer the per-language and per-transport axes from the SDK spec.

## Part 1 — Per-quadrant content requirements

What each quadrant MUST include, per CP-01 conventions.

### 1.1 Tutorials (MUST)

Tutorials hand-hold the reader from zero to first success. Conventions:

- **Per-language tutorial**: at least one per supported language (Python,
  TypeScript). Reader starts with no SDK installed; ends with a
  working call in a notebook or REPL.
- **Per-transport setup tutorial**: at least one per supported transport.
  Reader has the SDK but hasn't set up the transport yet. For HTTP it's
  "create an API key and authenticate"; for MCP it's "configure Claude
  Desktop / Cursor to launch the MCP server"; for subprocess it's
  "install the CLI binary and verify the version".
- **Length cap**: 10-15 minutes of reading and following along. Longer
  tutorials drop reader retention.
- **Single success path**: no branching. If the reader has a choice,
  pick one and document the alternatives in a how-to.

**Conventional file structure**:

```
docs/tutorials/
├── getting-started-python.md
├── getting-started-typescript.md
├── mcp-with-claude-desktop.md
├── mcp-with-cursor.md
└── using-the-cli.md
```

### 1.2 How-to guides (MUST)

How-to guides assume the reader knows what they want to do. They give
the recipe. Conventions:

- **Task-shaped titles**: "How to [verb] [object]". E.g., "How to handle
  rate limits", not "Rate limit handling".
- **Self-contained**: each guide is independent. Cross-link, don't
  duplicate, but don't assume the reader read another how-to first.
- **End in working code**: every how-to ends with a snippet the reader
  can copy. Bonus: a link to the snippet in a runnable repo.
- **One transport per guide when transport-specific**: don't try to
  cover "how to handle rate limits across HTTP and gRPC" in one doc;
  split.

**Minimum set of how-tos** (most SDKs need these regardless of domain):

| How-to | Why |
|---|---|
| Configure timeouts, retries, base URL | The configuration knobs in the SDK |
| Catch and recover from errors | The three categories + domain errors |
| Iterate over a paginated list | If pagination is exposed |
| Stream a long-running response | If streaming is exposed |
| Authenticate (per transport + per auth method) | Per Part 7 of the SDK spec |
| Access raw response headers / status | The escape hatch |
| Use idempotency keys | Where the SDK supports them |
| Use the MCP server | Per Part 4.2 of the SDK spec |
| Use the CLI from a script | If a CLI ships alongside |
| Handle timestamps / dates | Conventional types that the reader will hit |
| Mock the SDK for testing | Testing pattern the SDK supports |
| Use the SDK in a browser (TypeScript only) | If browser-compatible |
| Use custom HTTP transport (TypeScript) | `fetch` override patterns |

**Conventional file structure**:

```
docs/how-to/
├── configure-client.md
├── handle-errors.md
├── iterate-paginated-results.md
├── stream-responses.md
├── authenticate-http.md
├── authenticate-mcp.md
├── access-raw-response.md
├── use-idempotency-keys.md
├── use-mcp-server.md
├── handle-timestamps.md
├── mock-for-testing.md
├── browser-usage.md
└── custom-http-transport.md
```

### 1.3 Reference (MUST)

Reference is exhaustive, dry, authoritative. Conventions:

- **Generated from the source-of-truth schema** (per the SDK spec).
  Hand-written reference docs drift; generated ones don't.
- **Per-language**: same operations, but documented in each language's
  idiom (Python signatures with type hints; TypeScript signatures).
- **Per-operation**: every operation gets its own reference page (or
  section). Signature; required params; optional params; return
  type; possible errors with category + transport code; minimum
  working example.
- **Cross-linked from how-tos**: every how-to that mentions an
  operation links to the reference page for it.

**Required reference sections**:

| Reference | Source | Auto-generated? |
|---|---|---|
| API reference (Python) | OpenAPI 3.1 / Protobuf / Smithy / etc. | Yes |
| API reference (TypeScript) | Same schema | Yes |
| MCP tool reference (every tool) | Same schema → MCP `tools/list` | Yes (the MCP server `tools/list` IS the reference at runtime; docs version is the human-readable rendering) |
| Configuration reference | The codegen config | Yes |
| Error reference | The error model + transport bindings | Mostly (hand-write extensions for domain errors) |
| Type reference (all schemas) | The source schema | Yes |
| Transport binding reference (one per supported transport) | Part 4 of the SDK spec | Hand-write, generation-assisted |

**Conventional file structure**:

```
docs/reference/
├── api-python/                  # auto-generated
├── api-typescript/              # auto-generated
├── mcp-tools/                   # auto-generated
├── configuration.md             # auto-generated
├── errors.md                    # auto-generated + hand-written extensions
├── types.md                     # auto-generated
└── transport-bindings/
    ├── http.md
    ├── jsonrpc-mcp.md
    └── subprocess.md
```

### 1.4 Explanation (MUST)

Explanation is conceptual, why-shaped. Conventions:

- **Title is a question or a concept**: "Why we picked outcome-category
  errors", "How retries work under the hood".
- **Stand-alone-readable**: doesn't require reading the how-to or the
  reference first. An interested reader who wants to understand can.
- **Cites the SDK spec** where decisions were made. If the reader
  wants to push deeper, they have the path.
- **Includes the wart confessions** — the parts of the SDK that aren't
  ideal but the team accepts. Hidden warts erode trust; documented
  warts build it.

**Minimum set of explanation docs**:

| Explanation | Why |
|---|---|
| The mental model (operations + transport + schema) | Lifts Part 1 of the SDK spec to user-facing |
| Why the error categories are Protocol / Expected / Internal | Decision context behind the error model |
| Why retries default to N attempts with backoff (and when not) | Behaviour the reader will be surprised by |
| Why field case is preserved from the wire | The most common stumbling block in TypeScript |
| The async / sync split (Python) — when to use which | Decision guide |
| Known quirks and accepted compromises | The "identity-mismatch wart" and friends |
| Versioning policy and what counts as breaking | Sets expectations for upgrades |

**Conventional file structure**:

```
docs/explanation/
├── mental-model.md
├── error-categories.md
├── retry-strategy.md
├── field-case-conventions.md
├── async-vs-sync.md
├── quirks-and-compromises.md
└── versioning-policy.md
```

## Part 2 — Cross-cutting docs (orthogonal to quadrants)

A few categories sit alongside the four quadrants, not inside them.
They're orthogonal because they serve a different organising principle.

### 2.1 Migration notes (per release)

For each minor/major version bump, produce a migration note documenting:

- **What changed** (the diff in the public surface)
- **Breaking changes** explicitly called out
- **Auto-migration commands** if available (codemods, scripts)
- **Manual migration steps** when not automatable
- **Deprecation timeline** for soon-to-be-removed features

Conventional file: `docs/migrations/v{from}-to-v{to}.md`

This is technically Reference (per Diátaxis), but the per-release
slicing makes it operationally distinct. Worth its own folder.

### 2.2 Troubleshooting / FAQ

Symptom-first navigation: the reader has a problem and starts from
the error message or unexpected behaviour. The doc routes them to the
underlying cause.

Conventional structure:

```
docs/troubleshooting/
├── README.md            # Indexed by symptom
├── connection-refused.md
├── unexpected-401.md
├── rate-limit-loops.md
└── mcp-server-not-found.md
```

This is technically How-to (problem → solution), but starting from
symptoms is so different a navigation model from task-oriented how-tos
that the convention is to separate them.

### 2.3 Recipes / cookbook

Composed how-tos. Each recipe combines multiple operations to
accomplish a real-world end-to-end task.

Example recipes:
- "Build a CLI that watches a directory and calls the SDK on changes"
- "Build a webhook handler that retries failed deliveries"
- "Run the SDK from a Lambda / Cloud Function"
- "Compose two MCP servers behind a single client"

Conventional file: `docs/recipes/{recipe-name}.md`

This is how-to in spirit; the "cookbook" framing emphasises that
recipes are larger and combine primitives.

### 2.4 Changelog

The flat list of every change. The migration notes section is
opinionated about how to handle each change; the changelog is the
record.

Conventional file: `CHANGELOG.md` at repo root. Following
[Keep a Changelog](https://keepachangelog.com/) convention.

## Part 3 — The per-language and per-transport axes

Each quadrant interacts with two axes from the SDK spec:

| Axis | Affects |
|---|---|
| **Language** (Python, TypeScript, …) | Tutorials, How-tos that touch language idioms, Reference (per-language sections), Explanation (language-specific decisions) |
| **Transport** (HTTP, MCP, subprocess, …) | Tutorials (per-transport setup), How-tos (per-transport auth, telemetry, retries), Reference (per-transport binding details), Explanation (why this transport, when to use it) |

## Part 4 — Agent-consumable conventions for each doc

What makes documentation agent-consumable (not just human-consumable):

| Convention | Applies to | Why |
|---|---|---|
| **Copy-pasteable code blocks** | Tutorials, How-tos, Recipes | Agents pattern-match on whole snippets; partial snippets force the agent to invent the surrounding context. |
| **Type-information embedded** | Reference, How-tos | Agents reading "client.x.method(name)" without knowing the type of `name` have to guess. Always show types in signatures. |
| **Failure modes documented per operation** | Reference | An agent calling an operation needs to know what can go wrong so its error-handling code can route correctly. |
| **Cross-references to the schema** | Reference, Explanation | Agents that need to construct a request use the schema; docs that show "here's the field but no schema link" force re-derivation. |
| **Versioning visible in every doc** | All quadrants | Agents should know which version they're reading. Single header at top: "This doc applies to v{X.Y.Z}". |
| **Examples include error cases, not just success** | How-tos, Recipes | "Here's how to call X" should be matched by "here's what X looks like when it fails". |
| **Plain-English summaries at the top** | All quadrants | Per FE-04 (scannable in 30 seconds) — an agent skimming should know whether to read more. |
| **Stable URLs / anchors** | All quadrants | Agents bookmark / cite specific sections; URL drift breaks the citation. Don't rename anchors lightly. |
| **MCP tools also include human-readable descriptions in the schema** | Reference (MCP tool reference) | Per the SDK spec, the operation `description` is the LLM-facing prompt at invocation time. The reference renders it for humans + agents reading it offline. |

## Part 5 — Authoring and generation pipeline

How the docs get produced and kept current.

### 5.1 Generated vs hand-written

| Category | Generated from | Hand-written |
|---|---|---|
| API reference (per language) | OpenAPI / Protobuf / Smithy schema → docs generator | No |
| MCP tool reference | Same schema → MCP server `tools/list` output formatted as markdown | Description authoring (which is what the LLM sees) |
| Configuration reference | The codegen config | No |
| Type reference | The schema | No |
| Transport binding reference | Hand-written initially; assisted by the SDK spec's Part 4 | Yes |
| Error reference | Generated from the error model + hand-written domain extensions | Partial |
| Tutorials | Hand-written | Yes |
| How-to guides | Hand-written | Yes |
| Explanations | Hand-written | Yes |
| Migrations | Hand-written per release; can be machine-assisted (codemod docs) | Yes |
| Troubleshooting / FAQ | Hand-written; grown from real issues | Yes |
| Recipes | Hand-written | Yes |

### 5.2 Generation tools (per ecosystem)

Per CP-01, established conventions exist. Open-source choices:

| Doc category | Python tool | TypeScript tool | Universal |
|---|---|---|---|
| API reference from OpenAPI | [pdoc](https://pdoc.dev/) / [sphinx](https://www.sphinx-doc.org/) | [TypeDoc](https://typedoc.org/) | [Redoc](https://github.com/Redocly/redoc) or [Swagger UI](https://swagger.io/tools/swagger-ui/) for the schema directly |
| API reference from Protobuf | [protoc-gen-doc](https://github.com/pseudomuto/protoc-gen-doc) | Same | Same |
| Hand-written prose | [Sphinx](https://www.sphinx-doc.org/) | [TypeDoc](https://typedoc.org/) for combined; [VitePress](https://vitepress.dev/) for prose | [Docusaurus](https://docusaurus.io/), [MkDocs](https://www.mkdocs.org/) (with [Material](https://squidfunk.github.io/mkdocs-material/) theme), [VitePress](https://vitepress.dev/) |
| MCP tool reference | Generated from `tools/list` output via a small markdown formatter | Same | Same |

**The MkDocs + Material pattern** is the most common open-source choice
for SDK docs as of 2026. It supports per-language sections, search,
versioning, and integrates with auto-generated reference output. Other
defensible choices: Docusaurus (React-based; preferred when the SDK
ships JS-heavy); VitePress (lighter; faster); Sphinx (older;
deeper ecosystem but heavier configuration).

### 5.3 Per-version docs

Maintain docs per major version. Convention:

```
docs/
├── v1/
│   ├── tutorials/
│   ├── how-to/
│   ├── reference/
│   ├── explanation/
│   └── migrations/
├── v2/
│   └── ...
└── latest -> v2/      # symlink
```

A version dropdown at the top of the docs lets the reader switch. Old
versions stay readable; new versions are the default.

## Part 6 — Confidence calibration (CC)

Per the SDK spec's pattern, calibrate confidence per major claim:

| Claim | Tier | Reasoning |
|---|---|---|
| Diátaxis is the right primitive set for SDK docs | VALIDATED | Adopted by Django, GitLab, Cloudflare, GitHub, the Linux kernel. Multi-vendor convergence. |
| Every quadrant is MECE-required | SUPPORTED | The user's missing-docs table maps fully onto the four quadrants. |
| Per-language sub-organisation within each quadrant | SUPPORTED | Most multi-language SDKs (Stripe, Twilio, AWS) split docs by language. |
| Per-transport sub-organisation within each quadrant | EMERGING | Less established because most SDK docs are single-transport. Recommended on the basis that the SDK spec is multi-transport. |
| Auto-generation of reference docs | VALIDATED | The clear convention; hand-written reference docs drift. |
| Embedding type information for agent consumption | EMERGING | Not yet a published industry convention; recommended based on agent-tool usage patterns observed in MCP servers. |
| Worked recipes as a distinct cookbook category | SUPPORTED | Common pattern (Stripe, Twilio, AWS); name varies (cookbook, recipes, tutorials, examples). |

## Part 7 — Adversarial testing pass (AT)

Where would this docs spec fail?

1. **Diátaxis is famously misunderstood.** Authors mix categories
   accidentally — a "tutorial" that drifts into how-to or vice-versa.
   The discipline isn't in knowing the four exist; it's in keeping
   each one in its lane. Mitigation: Part 1 above lists the conventions
   per quadrant explicitly; the quality checklist in Part 9 enforces.

2. **The per-transport axis is overhead for single-transport SDKs.**
   If the SDK ships HTTP only, the `tutorials/mcp/` and
   `tutorials/subprocess/` folders are empty. Recommendation: don't
   create empty folders. Use the structure only as transport coverage
   grows.

3. **Generated reference can be lower-quality than hand-written.** If
   the schema authoring (per the SDK spec) is weak — empty descriptions,
   missing examples — the generated reference is also weak. Mitigation:
   the SDK spec's Part 13 quality checklist already requires LLM-grade
   `description` fields on every operation; this docs spec's Part 9
   adds a per-doc spot-check.

4. **The "agent-consumable" conventions in Part 4 are unvalidated.**
   Type-information embedding, plain-English summaries, schema
   cross-references — these are reasonable but not yet observed at
   scale. Mitigation: ship the recommendations as SHOULD, not MUST;
   collect feedback; tighten in a future revision.

5. **Versioning policy lives in two places.** Migration notes (Part 2.1)
   and the explanation doc on versioning (Part 1.4) overlap. The split
   we propose: explanation is the *policy* (what counts as breaking);
   migrations are the *per-version specifics*. Worth being explicit so
   they don't grow redundant.

6. **Cookbook and how-to overlap.** Where's the line between "how-to:
   handle errors" and "cookbook: build a webhook handler that retries"?
   Recommendation: how-to = single operation focus; cookbook = multi-operation
   end-to-end task. Crisp distinction in title and scope.

## Part 8 — Falsifiability criteria (FR)

This spec is in calibration. Stop / pivot / re-evaluate:

**STOP applying this spec if:**

- After producing two SDKs' worth of docs (one HTTP, one MCP-shaped), the
  Diátaxis split is consistently violated by authors → the framework is
  too abstract for our audience.
- The per-quadrant minimum content lists (Parts 1.1-1.4) prove
  insufficient — readers routinely report missing topics that the
  minimum sets didn't cover.

**PIVOT if:**

- Auto-generated reference docs are illegible (too verbose; too
  shallow). Adjust the generation conventions.
- The per-transport folder layout proves more confusing than helpful
  for single-transport SDKs. Switch to layered-as-needed.

**RE-EVALUATE if:**

- An external standard emerges that subsumes Diátaxis (unlikely; it's
  the established convention).
- Agent-consumable documentation conventions (Part 4) settle in
  industry. Update to reflect.

## Part 9 — Quality checklist

Before shipping a generated SDK's docs from this spec, verify:

### Coverage by Diátaxis quadrant
- [ ] At least one tutorial per supported language
- [ ] At least one tutorial per supported transport
- [ ] All "minimum set" how-tos (Part 1.2) present
- [ ] API reference auto-generated per supported language
- [ ] MCP tool reference auto-generated (if MCP is a supported transport)
- [ ] Configuration reference auto-generated
- [ ] Error reference present (auto + hand-written extensions)
- [ ] Transport binding reference per supported transport
- [ ] All "minimum set" explanations (Part 1.4) present

### Cross-cutting
- [ ] Migration notes for every release (or "no breaking changes" if so)
- [ ] Troubleshooting section indexed by symptom (≥3 entries)
- [ ] Cookbook with at least 2 end-to-end recipes
- [ ] Changelog at repo root following Keep a Changelog

### Per-doc agent-consumable conventions
- [ ] Every code block is complete (copy-pasteable; imports + setup included)
- [ ] Every operation reference shows the signature with types
- [ ] Every operation reference lists failure modes
- [ ] Every doc has a version header
- [ ] Examples cover failure cases, not just success
- [ ] Plain-English summary at the top of each doc

### Generation discipline
- [ ] Reference docs are auto-generated from the SDK source-of-truth
      schema (no hand-edited reference)
- [ ] Generation is part of the release pipeline (docs don't drift
      from code)
- [ ] Per-version docs maintained; old versions readable

### Discoverability
- [ ] Docs site has search
- [ ] Each section links to the others (tutorials → how-to →
      reference → explanation)
- [ ] Version dropdown visible
- [ ] Stable URLs / anchors (don't rename)
