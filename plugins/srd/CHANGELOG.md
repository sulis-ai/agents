## Cumulative history (through v1.22.0)

> Migrated from `plugin.json` description, 2026-05-23.

Requirements Analyst — facilitates building Software Requirements Documents backed by UML artifacts through guided conversation. Includes intent triage, disambiguation sweep, and adversarial sweep phases; reads .context/{project}/INDEX.md from sulis-context to apply Respect-Don't-Restate; hands architecture work over to SEA. Publishes marketplace-wide standards used by every plugin: Founder English (FE-01..FE-11), the Repository Contract Standard (RC-01..RC-13), the PR Hygiene Standard (PH-01..PH-08), the Change Work Standard (CW-01..CW-08), Engineering Principles, Git Workflow, Convention Preference, Audience-Adapted Framing, Executor Loop, and Red-Green-Blue. v1.19.x shipped FE evolution (anchor cases 1-5, FE-06 extension to all founder-facing surfaces including Agent-tool description parameters). v1.20.0: publishes Repository Contract Standard at plugins/srd/references/repository-contract-standard.md (RC-01..RC-13). The marketplace-wide specification of what a GitHub repository must look like before any Sulis agent operates on it: branching model, branch protections, merge queue config on dev (group size 5, concurrency 3, failure tolerance 3 per Shopify pattern), six required GitHub Actions workflows (branch-ci on pull_request, merge-queue-ci on merge_group, deploy-staging on push to dev, health-and-smoke on workflow_run, promote-dev-to-main on workflow_dispatch, release-prod on push to main), staging + production environments, per-env deploy tokens, repository settings (default dev / squash-only / linear history), v* SemVer tag protection, token scopes, CODEOWNERS, the agent arrival check (Step 0 before Step 1 worktree creation) emitting structured JSON with PASS/FAIL per rule, and the strict-mode refusal contract with founder-English translation template. Composes with GIT-01..GIT-10 (operationalises the two-branch model on GitHub) and EL-01..EL-08 (adds Step 0). Reference workflow YAML inline; bootstrap workflow specified for the companion wpx-arrival-check + wpx-bootstrap-repo scripts in the next sulis-execution release. Strict mode for v1: agents refuse non-conforming repos. v1.21.0: publishes the PR Hygiene Standard at plugins/srd/references/pr-hygiene-standard.md (PH-01..PH-08). The marketplace-wide specification of change-artifact shape — Scope (one logical concern per PR), Size (reviewable in one sitting; Google's 200/400 line thresholds as calibration starting points), Safety (high-risk surfaces — migrations, schemas, secrets, infra config, lock-file dominance — flagged regardless of total size), Completeness (code + tests + docs ship together). Honest synthesis of Google's eng-practices, SmartBear/Cisco 2006 review study (recency-capped), DORA change-failure-rate research, Squawk (Stripe Postgres migration linter), Strong Migrations (Andrew Kane), Fowler's Expand-Contract pattern, Conventional Commits, and Trunk-Based Development doctrine. Pre-flight CTS analysis passed: MECE (pairwise mutual-exclusivity verified, leg-test passed, so-what test passed); Primitive Grounding (irreducible at PR-review level); Balanced Investigation (counter-evidence section discloses Facebook big-diff practice + SmartBear recency caveat + splitting-theatre risk); Adversarial Testing (three rejected alternatives — extend MECE-3, fold into Quality lens, distribute across existing standards — documented with reasoning); Falsifiability (stop/pivot/re-evaluate criteria + pre-mortem); Confidence Calibration (each primitive SUPPORTED individually, synthesis UNVALIDATED until calibration data accrues). 90-day calibration window begins 2026-05-21. Anchor case PR-168 documented. Consumed by /code-review via SEA v0.14.0+ (CR-09); designed for extension to any future PR-touching skill via the PH-06 signal-table contract. v1.22.0: publishes the Change Work Standard at plugins/srd/references/change-work-standard.md (CW-01..CW-08). The marketplace-wide specification of how a change is bounded, branched, and lifecycled as a unit of work. A change is realised as a git branch (change/{primitive}-{slug}) with a dedicated worktree; all artifacts for that change — SRD.md, TDD.md, work packages, hardening deltas, code, tests, code-review reports, security findings — live on the branch; multiple changes run in parallel on separate branches with no cross-change coordination required (v1 explicit independence assumption per CW-07); merge to dev ends the change. Naming maps onto SEA's 22 change primitives (create / extend / refactor / strangle / harden / delete / etc.) for documentation value. Two-level worktree hierarchy: the change worktree at the top, executor WP worktrees nested inside (branching off the change branch, not origin/dev). Honest synthesis of OpenSpec changes/ directory (Fission AI), Conventional Commits, SEA's change primitives, and GIT-01..GIT-10. Pre-flight CTS analysis passed: counter-evidence disclosed (long-lived change branches diverge; trivial changes don't need the overhead), adversarial alternatives rejected (spec-as-unit, project-as-unit, per-skill branching, no-branching), falsifiability criteria + pre-mortem. 90-day calibration window begins 2026-05-21. Consumed by sulis-execution v0.12.0+ (the sulis-change tool) and sulis-concierge v0.2.0+ (change-aware initiation). Deferred to v0.2.0: change dependency graph; cross-project changes as first-class concept.
