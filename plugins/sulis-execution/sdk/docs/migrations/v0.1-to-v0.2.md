# Migrating from sulis-execution SDK v0.1.0 to v0.2.0

**Released:** 2026-05-22

## What changed

v0.2.0 is a naming sweep â€” every method name is now self-describing
without insider knowledge. The behaviour is identical to v0.1.0; only
the public surface names changed.

Underlying CLI binaries and subcommands are unchanged (`wpx-pipeline`,
`wpx-step12`, `wpx-wp`, etc. â€” those keep their names). Only the SDK's
Python / TypeScript / MCP surface renames.

## Why

A peer SDK following the same spec hit usability problems in a real
test session: method names like `step12.wrap` and `wp.read_frontmatter`
required insider context to understand. The validation rubric was
extended to catch this class of issue (rubric v0.2.0); applying those
checks to this SDK surfaced 9 method renames + 2 resource renames.

## Migration: per-rename before/after

### Resources

```diff
- client.step12       # opaque (Step 12 of the executor lifecycle)
+ client.lifecycle    # self-describing
```

```diff
- client.wp           # "wp" is jargon
+ client.work_package # explicit
```

## What does NOT change

- **Underlying CLI tools** (`wpx-pipeline`, `wpx-step12`, `wpx-wp`,
  etc.) â€” keep their names. The SDK wraps them under cleaner public
  names; the CLI surface is unchanged.
- **The `wp` parameter** when passing a WP ID â€” still `wp='WP-001'`.
  That's the canonical input convention across the marketplace.
- **Wire field names** (e.g. `auto_draft_wp_id` in
  `FindingsDraftRemediationResult` â€” that's the JSON envelope field
  from the underlying CLI; preserved per parity contract).
- **Error class hierarchy** â€” `SulisExecutionError`, `ProtocolError`,
  `ExpectedError`, `InternalError`, `BinaryNotFoundError`,
  `InvalidArgumentError`, `UnexpectedOutputError` â€” all unchanged.
- **Transport layer behaviour** â€” exit-code mapping, correlation IDs,
  envelope parsing â€” unchanged.
- **All other operations** (pipeline.run, train.*, blocker.write,
  blocker.archive, worktree.create, worktree.remove, all change.*,
  index.flip_status, index.set_status, index.list_ready,
  index.read_config, journal.init, journal.start_step,
  journal.complete_step, journal.record_attempt,
  journal.record_preflight, journal.add_plan_item, journal.read,
  findings.register) â€” unchanged.

## Migration tooling

No automated codemod is provided. The renames are mechanical
(grep + replace your call sites). For a Python codebase:

```bash
# Per-rename sed expressions (run carefully; review diff before commit)
sed -i.bak 's/client\.step12\.wrap/client.lifecycle.complete/g' **/*.py
sed -i.bak 's/client\.wp\.read_frontmatter/client.work_package.read_metadata/g' **/*.py
sed -i.bak 's/client\.wp\.append_evidence/client.work_package.append_evidence/g' **/*.py
sed -i.bak 's/client\.index\.add_wp/client.index.add/g' **/*.py
sed -i.bak 's/client\.index\.propagate_blocked/client.index.mark_downstream_blocked/g' **/*.py
sed -i.bak 's/client\.index\.sync_auto_drafts/client.index.register_pending_drafts/g' **/*.py
sed -i.bak 's/client\.findings\.auto_draft_wp/client.findings.draft_remediation/g' **/*.py
sed -i.bak 's/client\.journal\.seed_plan/client.journal.create_plan/g' **/*.py
sed -i.bak 's/client\.journal\.mark_plan_item/client.journal.update_plan_item/g' **/*.py
sed -i.bak 's/client\.journal\.record_postdeploy/client.journal.record_security_verdict/g' **/*.py
```

For TypeScript, replace the camelCase versions.

## Compatibility

There are NO compatibility shims in v0.2.0. The old names are removed,
not deprecated. Pre-1.0 SemVer allows this; no external consumers
were registered when this rename was applied.

If you need v0.1.0 behaviour, pin to `sulis-execution==0.1.0` (or
`@sulis-ai/execution@0.1.0`); the package will continue to exist on
PyPI / npm.
