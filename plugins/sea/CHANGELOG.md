## Cumulative history (through v0.20.0)

> Migrated from `plugin.json` description, 2026-05-23.

Polyglot 4-stage workspace enumeration adds auxiliary packages, code-bearing dirs, and deployment-only dirs alongside the existing monorepo manifests. Outputs structured JSON + Markdown + a self-contained navigable HTML report. 88+ unit + integration tests; no LLM fallback. Every WP carries a change primitive (22-primitive Minto-pyramid catalogue). v0.11.6: cites Founder English (FE-01..FE-10) at plugins/srd/references/founder-english.md — every founder-facing chat message AND founder-readable artifact write passes the FE-06 five-point check (strip internal IDs, translate filenames, expand acronyms, strip internal taxonomy, read-aloud test). Lead with outcomes; concrete over abstract; no mechanism narration. v0.11.7: adds FE-11 (Inference Over Interrogation) citation — infer from existing context before asking the founder; never relay a specialist's 'open questions' verbatim. v0.12.0: adds /code-review skill — a PR-scoped umbrella that runs three lenses in parallel (architecture / security / general quality) against a PR number, branch ref, or commit range. Scans the diff plus a one-hop neighbour ring (callers, callees, shared-import modules) capped at 20 files; findings in the diff can block merge, findings in the neighbour ring are surfaced as exposure (downgraded one severity, never merge-blocking). Invokes /sulis-security:codebase-assess in Quick mode for the security lens. Outputs a single merged report at .architecture/{project}/code-reviews/PR-{number}-{date}.md plus draft Hardening Deltas with source: code-review:PR-NNN. Advisory only — never posts to the PR, never sets status checks, never auto-blocks. v0.15.0: rewrites /code-review report format to a two-tier shape per user feedback that the report is read by non-technical founders (often using AI to generate code) and currently fails FE-06 — internal IDs (CR-NN, PH-NN, HD-NNN), engineer jargon (circuit breaker, resilience policy, atomicity), and raw severity labels (critical/high/medium/low) make the report unreadable to its primary audience. Tier 1 (top of report) is founder-facing — plain English sections (At a glance / What to fix / How this pull request is shaped / Things to take away), action-oriented verdict labels (Ready to merge / Approve, but apply small fixes first / Needs changes before merge / Don't merge yet) and severity labels (Must fix / Strongly recommend fixing / Worth fixing / Minor), recommendations cite codebase patterns by file path, and an optional 'Things to take away' section (max 3 lessons, tied to specific findings, anti-condescension rules — omit when the PR is clean). Tier 2 (below ## Technical detail heading) is unchanged for engineers and downstream agents — CR-NN / PH-NN intact, signal tables, CR-08 self-attestation. Chat summary shortened to 3-4 sentences since the report itself is now founder-readable. v0.16.0: generated filenames across SEA (code-review reports), sulis-security (viability reports), IDC (journal entries), and sulis-product-development (validator reviews) now use ISO 8601 UTC compact timestamps (YYYY-MM-DDTHHMMSSZ) instead of date-only stamps. Closes user feedback that same-day reruns of any of these skills collide. Convention applied at filename level; document metadata fields (e.g. SRD 'Date:' headers) keep date-only granularity. Bash snippet documented in each affected skill: TIMESTAMP=$(date -u +%Y-%m-%dT%H%M%SZ). v0.17.0: /code-review now writes a self-contained per-review bundle at .architecture/{project}/code-reviews/PR-{number}-{TIMESTAMP}/ containing REVIEW.md + hardening-deltas/ subdir (with INDEX.md + HD-NNN files at status: proposed) + signals.json (machine-readable PH-06 signal table) + tool-outputs/ (raw scanner / typecheck logs). Closes user feedback that each review should be a self-contained directory rather than scattered files. /sea:harden updated to discover HD-*.md files recursively under .architecture/{project}/ instead of only the canonical flat hardening-deltas/ — picks up code-review drafts once their status is promoted from 'proposed' to 'accepted'. references/hardening-deltas.md documents both valid locations (canonical flat for audit/security/MUC/NFR sources; per-review bundle for code-review drafts) and notes that HD IDs remain unique per-project regardless of location. v0.18.0: adds /sea:suggest-split skill — natural follow-on to /code-review when PH-01 Scope or PH-02 Size fires high. Reads the diff, categorises changes by file pattern (migration / schema / test / infra / docs / lockfile / refactor / feature), proposes a 2-4 way split with dependency ordering, and emits the exact git commands (using checkout -b + checkout HEAD -- files pattern; safer than cherry-pick for non-experts) to make each new branch. Read-only by design — does NOT execute the commands. Output is a two-tier SPLIT-SUGGESTION.md written inside the originating review bundle. Designed for non-technical founders who get a high-severity hygiene finding and don't know what 'split the PR' actually involves. /code-review's Tier 1 report gains a 'How to split this pull request' subsection when PH-01 or PH-02 high fires, plus a chat-summary follow-on offer pointing at /sea:suggest-split. v0.2 of suggest-split could add /sea:apply-split for opt-in execution; out of scope for this commit. v0.19.0: Code Review Standard v0.2.0 adds CR-10 (Performance Procedural Checks) and the sibling document plugins/sea/references/performance-procedural-checks.md cataloguing 10 mechanically-detectable performance anti-patterns: N+1 DB query, N+1 RPC/HTTP, N+1 filesystem read, O(N²) over same collection, synchronous waterfall, unbounded materialisation, repeated invariant computation in loop, wasted DB roundtrips, string concat in hot loop, scan-heavy filter on non-indexed column. Each pattern has a regex+filetype signature, severity default per CR-05, evidence template citing CR-04, and false-positive guidance. /code-review's Quality lens (CR-07) gains a 7th mandatory output: CR-10 procedural scan with file:line + quoted-text findings. CR-08 self-attestation checklist extends by one row. CR-10 closes a real gap: a live executor session composed two WPs in parallel (one added get_user(id); another added a loop calling it) that together became an N+1 query in production. Neither WP in isolation looked wrong; the LLM quality lens didn't flag the composition. Mechanical signatures catch this class deterministically. 90-day calibration window from 2026-05-22. Triggered by founder feedback wanting performance anti-patterns caught at the code-review layer. v0.20.0: adds the Decompose Validation Rubric at plugins/sea/references/decompose-validation-rubric.md (DV-01..DV-NN across 6 phases, ~30 checks). Run automatically by /sea:decompose at end of workflow (step 11 — Validate); produces DECOMPOSE_VALIDATION.md alongside INDEX.md. Phases: P1 Inventory completeness (every WP has Context/Contract/DoD/Sequence/Token cost/Dependencies), P2 Atomicity (≤15 files MUST, ≤8 SHOULD, no `and` in titles), P3 Module naming + clean code (no jargon prefixes, no abbreviations, descriptive slugs), P4 Dependency graph correctness (no cycles, valid topological order, transitive depth ≤8), P5 Performance + non-functional reqs (endpoint/handler WPs declare measurable performance bounds), P6 Peer-collision risk (no two WPs Create the same file — catches the platform-repo loader/__init__.py collision class at breakdown time). Composes with the Code Review Standard CR-01..CR-10 (pre-execution analog to post-execution review) and with Founder English FE-01..FE-11 (verdict + blocking gaps pass FE-06; rubric body is internal taxonomy). v0.14.0: adds CR-09 (PR Hygiene Application) to the Code Review Standard and rewires /code-review to apply the new PR Hygiene Standard (PH-01..PH-08) published by SRD v1.21.0. Designed for extension to any future PR-touching skill via the PH-06 signal table contract. v0.13.0: publishes the Code Review Standard at plugins/sea/references/code-review-standard.md (CR-01..CR-08).
