# Glossary — Sulis app: drive a change from the app

The locked vocabulary for this specification. All artifacts use the preferred terms.

| Term | Definition | Also known as | NOT the same as |
|---|---|---|---|
| **Change** | A unit of work, modelled as a navigable lifecycle thread. The core primitive of the whole product. | — | A file (Cursor's unit); a pull request |
| **Board** | The landing view: every in-flight change shown as cards in columns, one column per stage. | Kanban, landing | The sidebar (a flat list, exists already) |
| **Stage** | Where a change sits in its lifecycle: recon → specify → design → implement → review → ship. | Lifecycle stage | "Status" (a plain-English read-out, see below) |
| **Thread** | One change's detail view inside the app: stage track, status, brain, files, chat. | Change thread, per-change view | The conversation alone (one part of the thread) |
| **Conversation** | The chronological chat history (founder prompts + agent replies) for a change. | Transcript, chat history | The "status" read-out |
| **Status** | The plain-English "what's happening" read-out for a change, computed at read-time. | Progress read-out | A periodically auto-published checkpoint (a non-goal here) |
| **Session** | A running `claude` process for a specific change. | Agent session | The change (data); the conversation (its output) |
| **Liveness** | Whether a change's session is currently running. Detected by a side-effect-free probe. | Running indicator | Reachability for chat (a stricter check) |
| **Brain** | The entities and workflows the agent has created for a change. | — | The conversation; the worktree files |
| **Entity** | One brain item (e.g. a requirement, a workflow step) with content. | Brain item | A file |
| **Worktree** | The on-disk folder of files for a change, kept tucked away under `~/.sulis/changes/`. | — | The change record |
| **Rendered preview** | A file (e.g. `.md`, `.html`) shown the way it is meant to look, not as raw source. | Preview | The raw source view (its toggle) |
| **The seam** | The single API boundary the app reaches all its data through; never the filesystem directly from the client. | Data seam, API boundary | A cloud service (the seam can later be served from the cloud, but is not one now) |
| **Relay** | The server path that delivers a founder's message to a session and streams the reply back. The app's only sanctioned write path (it includes the resume/spawn it triggers). | Chat relay | A read endpoint |
| **Resume** | Restarting a change's most recent (closed) session from its persisted conversation transcript, so the agent wakes with full prior memory. A built-in capability, not invented here. | Resume session | Spawn (which starts a brand-new session) |
| **Spawn** | Starting a fresh session for a change that has never had one, grounded in the change's saved context (CONTEXT.md / manifest / stage / prior decisions) so the agent re-reads the change before acting. | Spawn session | Resume (which restarts an existing session) |
| **Recoverable state** | What a resumed session can restore: the persisted transcript + the worktree files on disk + the brain entities the session created. A step interrupted mid-action at close is re-run, not magically continued. | — | A live in-memory session (lost at close) |
| **Needs attention** | A change that is blocked, waiting on a founder decision, or whose agent stopped mid-reply. An idle-but-fine change is not flagged. | — | "Not running" (a liveness state) |
| **Discovery agent** | A headless agent (`claude -p` / discovery agent) driven over the **same stream bridge** as the two-way chat, used for cold-start onboarding and starting changes from intent. | Headless agent, onboarding agent | The two-way chat *session* (same bridge, different job) |
| **Cold-start onboarding** | The first-run flow: a conversation that *creates* the entity graph (Tenant/Product/Project) when there's nothing to pick from yet. | Onboarding | A "pick Product → Project" form (useless against an empty graph) |
| **Spine emitter** | A schema-validated writer that creates a core entity (Tenant / Product / Project). Onboarding mints only through these. | Entity emitter | A freehand file write (bypassing validation) |
| **Tenant / Product / Project** | The core graph entities discovery mints. **Project** carries `source = {repo, path, primary_branch}`, which the app maps to the change machinery's `--repo-root`. | The graph, the spine entities | A change (a unit of work *within* a Project) |
| **Change primitive** | The kind of a change, picked from the change-primitives vocabulary by the existing classifier from the founder's plain-English intent. | Primitive, change kind | The change slug (its name/handle) |
| **Local-first** | The rule that a Project's repo must be present on the machine to start a change against it; if absent, the app clones from `Project.source.repo` first. | — | Cloud-hosted (a non-goal here) |
| **Concierge** | The chat front door, modelled on the `sulis:sulis` agent. It **coordinates only — it never does the work itself**: it sets you up (onboarding), starts changes, and answers read-only questions about your world (find changes, report status, Q&A). Its only direct consequential acts are minting setup entities and creating a change. | Chat front door | The two-way chat *to a change's session* (the concierge delegates work *into* a change; the session does the work) |
| **Containment rule** | The load-bearing principle that **all real activity — including investigation — happens inside a change**, never inline in the concierge. Even exploring something triggers the creation of a change to hold it, so the activity is audited, self-contained, and can evolve straight into a build. "The change is the unit of everything," extended to investigation. | "The change is the unit of everything" | The confirm gate (the gate is *when* the concierge may act; containment is *where* work happens) |
| **Investigation** | Exploring / looking into something — real activity, not a read-only lookup. By the containment rule it is held in its own **change**, not done inline in the concierge. | Exploration | A read-only concierge question (which does no work and creates nothing) |
| **Active Product** | The one Product the board (and the rest of the per-product views) is currently scoped to. A Tenant may have many Products, but only one is active at a time; its in-flight changes are what the board shows. | Current product, scoped product | The Tenant (which owns many Products); a Project (which lives under a Product) |
| **Product switcher** | The control that lets the founder change which Product is active. Switching it re-scopes the board — and the rest of the per-product views — to the newly chosen Product. | Product selector | A filter (which narrows *within* the active Product's changes; the switcher changes *which* Product) |
| **Repo find-or-create** | The onboarding branch on "where are your repos?": if the founder has no repo, the agent **creates** one for them (a consequential, possibly-external act, behind the confirm gate); if repos exist, the agent **finds/configures** them from the founder's pointer (bounded to the chosen area). | Find-or-create, repo branch | The local-first clone (which fetches an *already-known* `Project.source.repo`; find-or-create establishes that repo in the first place) |
| **Persisted Product/Project config** | The durable, schema-valid Tenant/Product/Project graph onboarding leaves behind — including each `Project.source = {repo, path, primary_branch}`. Because it persists, re-opening the app and saying "make a change on Product X" starts a change without re-running setup. | Durable setup, saved config | A live session (lost at close); a one-off in-memory onboarding answer |
