# GLOSSARY — auto-back-merge-on-release

The vocabulary locked for this specification. Every recurring noun in SRD.md,
NFR.md, MISUSE_CASES.md, and the diagrams reconciles against this glossary.

## Preferred Terms

| Term | Definition | Also Known As |
|------|------------|---------------|
| **back-integration** | The act of bringing main's post-release state (bumped version files + deleted changesets) back onto dev so dev's HEAD is at or beyond main's HEAD. Today done manually; this specification automates it. | back-merge (as a noun); reconcile-dev |
| **back-merge** | The specific merge or fast-forward operation that performs a back-integration. When used as a verb, "back-merge dev to main" means "advance dev to match main." | — |
| **back-merge PR** | A pull request whose only purpose is to back-integrate main into dev. Opened by the release robot when a clean fast-forward is not possible (the raced case). Distinct from a normal feature PR — its base is `dev`, its head is `main`, and its body is generated by the workflow. | recovery PR |
| **clean release path** | The expected case: between the moment the release PR was opened and the moment the release robot finishes its bump-and-tag work on main, dev has not received any new commits. The robot can fast-forward dev to main with no risk. See UC-001. | happy path |
| **drift** | The state where dev's `plugins/sulis/.claude-plugin/plugin.json` version, marketplace.json metadata.version, or `.changesets/` contents do not match main's. Drift is detectable; the most reliable check is `git merge-base --is-ancestor main dev`. | dev-behind-main; stale-dev |
| **drift detection** | A defensive check that compares dev's state against main's and refuses to proceed if dev is behind. Lives in two places: `/sulis:release-train` (refuses to draft a release PR against a stale dev) and `/sulis:change start` (refuses to start a new change on a stale dev). See FR-009, FR-010. | dev-behind-main check |
| **fast-forward** | A git operation that advances a branch pointer to a newer commit without creating a merge commit, only valid when the target branch's current commit is a direct ancestor of the new commit. The clean release path uses fast-forward; the raced path cannot. | ff; ff-only |
| **fork-consumer** | A repository that has installed the Sulis plugin (via the marketplace registry) and is using its release-train workflow. The marketplace itself is a fork-consumer of its own plugin (n=1 dogfood). | downstream consumer; plugin installer |
| **race window** | The time interval between (a) the release PR being opened against a known dev SHA, and (b) the release robot running its bump-and-tag work on main. Any commit landing on dev inside this window invalidates the fast-forward and forces the raced release path. | race interval |
| **raced release path** | The exception case: a commit landed on dev during the race window, so the robot cannot fast-forward dev to main. The robot opens a back-merge PR instead and never force-pushes. See UC-002. | raced path |
| **release robot** | The GitHub Actions job that runs on push to main (the `release-on-merge` workflow). Operates as `github-actions[bot]`. Today: reads pending changesets, bumps versions, tags, pushes to main. Extended: also performs the back-integration step. | the bot; the robot |
| **reusable workflow** | The GitHub Actions workflow file that lives in the plugin (`plugins/sulis/templates/workflows/release-on-merge.yml`) and contains the release-and-back-merge logic. Fork-consumers reference it via a `uses:` directive. | central workflow; canonical workflow |
| **shim** | A short workflow file (~10 lines) in a fork-consumer's `.github/workflows/release-on-merge.yml` whose only job is to call the plugin's reusable workflow with `uses:`. The shim is the consumer's opt-in. Deleting the shim opts out. | shim workflow; consumer shim |
| **version-pin** | A consumer's shim references the reusable workflow at a specific plugin version tag (e.g., `@sulis-v0.86.0`). Pinning protects against breaking changes; opt-in always-track (`@dev`) is permitted but documented as the riskier choice. See FR-006. | semver-pinned reference |

## Disambiguation — NOT the Same As

| A | B | Distinction |
|---|---|-------------|
| **back-merge PR** | back-integration commit | A back-merge PR is a recovery mechanism (raced case); a back-integration commit is the artifact left on dev by the fast-forward (clean case) OR by the merge of the back-merge PR (raced case). The commit appears either way; the PR only appears on the raced path. |
| **reusable workflow** | shim | The reusable workflow lives once, in the plugin. The shim is the consumer-side thin file that calls it. Multiple consumers, one reusable workflow. |
| **drift** | release in progress | Drift is a *persistent* state of dev being behind main. A release-in-progress is the *transient* race window. Drift detection should not false-positive during a normal release; it fires only when a release completed but back-integration didn't. |
| **fast-forward** | merge | Fast-forward leaves no merge commit; merge creates one. The clean release path uses fast-forward (dev is updated by reference move only); the raced path opens a back-merge PR which, on merge, creates a merge commit on dev. |
| **fork-consumer** | the marketplace itself | The marketplace IS a fork-consumer of its own plugin (it shims to its own reusable workflow). When this spec says "fork-consumer," it includes the marketplace unless the context is specifically about external installations. |

## Out-of-Scope Vocabulary

These terms appear in the change brief but are deferred to follow-on work; they
are documented here so future readers know they exist but should not be specified
inside this SRD:

- `/sulis:bootstrap-workflows` — a hypothetical skill that would install shim
  files into a fresh fork-consumer. Out of scope; surfaced as Recommended
  Follow-on in HANDOVER.md.
- discovery extension — extension of `/sulis:discover-project` to detect missing
  or stale shim files. Out of scope; surfaced as Recommended Follow-on.
- branch protection auto-setup — automated configuration of `dev` and `main`
  branch protection rules via the GitHub API. Out of scope; depends on access
  patterns we don't have.
