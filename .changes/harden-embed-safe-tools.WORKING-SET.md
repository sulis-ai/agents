## 1. Problem  (→ Opportunity)
Embed shipped L1 (safe-fetch proxy) + L2 (scoped file-tools) into how the agent operates so they are actually used (not bypassed by raw WebFetch/Read/Write/Bash) — via harness enforcement, not prose. Decide the embedding approach + exactly how far the Claude Code harness can enforce before/around the OS sandbox.

## 2. Current best solution  (→ Design)
A **tiered stack keyed to enforcement-locus** (NOT a single convention-vs-enforcement pick — that fork is a false trichotomy). Enforcement-locus is the primitive: every rule is adjudicated at exactly one of (i) **model judgment** (prose — advisory, bypassable), (ii) **Claude Code's tool adjudicator** (permission deny-rules + PreToolUse hooks — deterministic + hard-blocking for *recognized direct tool calls*; model-unbypassable under managed settings), (iii) **OS kernel** (the shipped Seatbelt/bubblewrap sandbox — covers *all* processes incl. spawned subprocesses). The plan, cheapest→strongest:
1. **Permission deny-rules** block raw network/file tools (`WebFetch`, `Bash(curl…/wget…)`); allow only the safe-fetch tool. (Locus ii. Note: arg-matching on Bash is fragile per Anthropic docs — block the *tools*, lean on the sandbox for the rest.)
2. **PreToolUse hook = the #130-grade structural guard:** match `Write|Edit|Bash`, read `tool_input.file_path`/`command`, run #130's `within_change_scope`, return `deny`/exit 2 for out-of-scope. Hard-blocks; overrides allow-rules. (Locus ii.)
3. **Enable the SHIPPED OS sandbox** (`sandbox.enabled`, narrow `allowedDomains`, `denyRead` creds; consumer-side managed `failIfUnavailable`+`allowUnsandboxedCommands:false`). Only this catches subprocess bypass (`python -c 'open()'`, obfuscated curl). (Locus iii.)
4. **Prose (convention) ONLY for quality** — "prefer safe-fetch for clean/low-token output" in sulis.md/agent-defs/skills. A bypass there costs polish, not safety.
Discipline that prevents false-safety: **every embedded rule is labelled by its enforcement-locus + threat-scope** (accidental-closed-now vs adversarial-deferred). Grounded in 4 critical-thinking spirals: 01KW3BZNS… → 01KTZRTN0… → 01KTZTCSP2… → 01KV0G2GCKXZR82C03WHPA17XD.

**Allowed-write-roots are CONFIG-DERIVED + a single source of truth shared by L2 (file-tools) AND L3 (sandbox).** Load-bearing because the brain is moving OUT of the worktree (v0.153.0 configurable brain_base_dir): the agent writes to the brain constantly, so its resolved path MUST be a writable root or the agent breaks. Rules: (a) resolve the brain path via the SAME `brain_base_dir` resolver (explicit → SULIS_BRAIN_BASE_DIR → repo-contract → default) — never hardcode `~/.sulis`; (b) ONE resolver computes the write-roots; both the file-tools scope check and the sandbox writable-path config read it (no drift); (c) **narrowest root that works** — allow the specific resolved brain subtree, NOT the whole `~/.sulis/` (which holds OTHER changes' state + sockets + cockpit cache → allowing it reopens the cross-change contamination #130 closed); (d) two root *kinds* — change-scoped (worktree + this change's state: only-mine) vs shared (brain: rw, legitimately shared across changes via provenance stamping; tools/cache: ro); (e) extra routes over time = reviewed CONFIG additions, each checked against narrowest-root, not code edits.

- **D6 — Embedding has a PREREQUISITE "Step 0": expose the safe tools as MCP tools + tighten the agent tool-allowlist.** Q1 check found L1/L2 are Python LIBRARIES, not denyable tool identities; the Sulis agent runs `tools: "*"` (all built-ins) and NO Sulis MCP server exists (`.mcp.json` = mobbin + playwright only). So "allow safe / deny raw" is not expressible until safe-fetch + the four file-tools are registered as MCP tools and `tools:"*"` is narrowed. NUANCE: L1's *enforcement* isn't "deny WebFetch / allow a tool" — it's the egress-proxy + sandbox FORCING raw outbound through the proxy (the agent uses raw HTTP under the hood, intercepted via `SULIS_SAFE_FETCH_PROXY`); the MCP tool is about making the safe path *available + ergonomic*. L2's enforcement is the `within_change_scope` PreToolUse hook on Write/Edit/Bash. So: MCP exposure = the *safe path*; hook+sandbox = blocking the *unsafe path*. **PACKAGING (answered): the safe-tools MCP is a PYTHON stdio server launched via the plugin's own Python env (uv/python), shipping WITH the plugin (scripts/ + registered in the plugin `.mcp.json`) — NOT npx. npx is only for external Node servers (playwright). Precedent: the prior `sulis-execution-mcp` (Python/PyPI) + plugin-builder `create-tool` (produces a Python MCP server).** Status: **accepted.**
- **D7 — The writable-root inventory (Q5) is RESOLVER-COMPUTED at spawn, narrowest-root, confirmed set.** (a) change-scoped rw: `~/.sulis/changes/{THIS_ID}/` (state.json, change.json); (b) shared rw: the *resolved* brain dir — and the DEFAULT brain is in-worktree (`<repo>/.brain/instances`) needing NO extra allow; only a *relocated* brain (e.g. `~/.sulis/brain`) adds a root; (c) runtime: `~/.sulis/session-manager.sock` + daemon log (daemon-owned); (d) read-only: plugin cache, git-common-dir, tools. Anti-pattern confirmed-rejected: allowing all of `~/.sulis/`. Status: **resolved → folds into D5.**
- **D8 — NOT a CLI→MCP migration: a bounded carve-out + a governance hook (spiral 01KV0KGGQQ…).** The right primitive is TWO orthogonal axes: **invocation-substrate** (raw-Bash / CLI / MCP) × **governance** (ungoverned / hook-governed / permission-denied) — NOT "what it touches". **Governance ≠ MCP**: a PreToolUse hook governs the existing CLIs without converting them, so "needs governing" never forces MCP. **MCP criterion (narrow):** MCP-ify iff *(typed/structured contract needed* **OR** *denyable identity at a trust boundary needed)* **AND** it doesn't bloat the selection surface; default = CLI. Binding MCP constraint = tool-**selection accuracy** (degrades ~30–50 tools + similar-name families), NOT token cost (Tool Search Tool, default Jan-2026, killed that). Concrete classification: **MCP now** = safe-fetch + scoped-file tools ONLY (the carve-out — their denyable identity is what makes "allow-safe/deny-raw" expressible); **stays CLI** = all ~55 sulis-*/wpx-* incl gh/git-push (trained-on, already ship-gated); **hook-governed** = one PreToolUse hook (allow `Bash(sulis-*:*)`/`Bash(wpx-*:*)`, flat-deny raw curl/wget; optional #130 scope-check on emit-* writes outside the worktree); **NEVER** = 1:1 MCP-ify the 21 emit-* (name-collision trap — if ever, ONE `sulis_emit(entity_type,payload)`). Honesty: MCP-identity is necessary-NOT-sufficient; the sandbox is the adversarial backstop (don't claim MCP = enforcement). Status: **proposed (answers the founder's "are we starting a CLI→MCP migration?" — NO).**

## 5. Rejected so far  (→ Decision.rejected_alternatives)
- **Pure-convention embedding (prose-only) as a safety boundary** — REJECTED. This repo's #211 prose warning failed twice before #130 replaced it structurally; instruction-only embedding leaks. Convention survives only for *quality* surfaces (D4).
- **Treating "L3" as a far-off bespoke build** — REJECTED/REFRAMED. The OS sandbox already ships in Claude Code; enabling it closes the subprocess gap now. Only the TLS-aware egress proxy is genuinely still to build.
- **Treating the shipped sandbox as a COMPLETE boundary** — REJECTED. Its proxy doesn't do TLS inspection → deliberate exfil via a *permitted* broad domain (GAP-β) survives. It closes accidental over-reach (GAP-α, the #211-shaped real threat) decisively; GAP-β stays deferred. Never claim "enforced" beyond the locus that delivers it.
- **Bash arg-matching deny-rules** (`Bash(curl <url>*)`) as the network control — REJECTED per Anthropic docs (fragile; bypassable by option-ordering/redirects/env-runners). Block the network *tools* + lean on the sandbox instead.
